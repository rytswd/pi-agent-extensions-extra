#+title: Telegram Bridge
#+state: complete
#+FILETAGS: :extension:feature:architecture:

* Summary

A persistent Go server (=pi-bridge=) that acts as the sole Telegram Bot API consumer, dispatching incoming messages to the correct pi session via localhost HTTP + SSE. Solves the multi-session polling race condition.

Single static binary, zero external Go dependencies (stdlib only: =net/http=, =encoding/json=, =sync=). 9.2MB binary, Nix-packageable.

* Motivation

Direct Telegram polling from each pi session is destructive — =getUpdates= delivers each update to only one consumer. When multiple sessions share a bot token, messages are lost or misrouted.

* Implementation

** Architecture

#+begin_example
┌──────────────────────────────────────────────────────┐
│  pi-bridge (Go binary, persistent)                   │
│  Config: ~/.config/pi-bridge/config.json             │
│  PID:    ~/.config/pi-bridge/bridge.pid              │
│                                                      │
│  Telegram Bot API ←→ Long-poll goroutine (single)    │
│                                                      │
│  Pi sessions ←→ HTTP API (127.0.0.1:PORT)            │
│    POST /register   { sessionId, cwd, adapter, topicId }     │
│    POST /unregister { sessionId }                    │
│    POST /send       { adapter, topicId, text, ... }  │
│    GET  /events     { sessionId } → SSE stream       │
│    GET  /health     → 200 OK                         │
│    GET  /sessions   → list registered sessions       │
│                                                      │
│  Routing table (in-memory):                          │
│    topicId=11 → session "abc" (pi-agent-extensions)  │
│    topicId=12 → session "def" (chronoa)              │
│                                                      │
│  No session for topic → Telegram reply:              │
│    "⚠️ No pi session connected."                     │
│  Duplicate topic → 409 Conflict (rejected)           │
└──────────────────────────────────────────────────────┘
#+end_example

** Adapter Pattern

The bridge uses an =Adapter= interface for extensibility:

#+begin_src go
type Adapter interface {
    Name() string
    Start(router *Router)
    Stop()
    Send(topicID int, text string, opts SendOpts) (messageID int, err error)
    React(messageID int, emoji string)
}
#+end_src

Currently one adapter: =adapter/telegram/=. Future adapters (Discord, Slack, Matrix) implement the same interface and register in =main.go=.

** Process Lifecycle

- Spawned by =telegram-connect= extension as detached process (=child_process.spawn= with =detached: true, stdio: 'ignore'=).
- Writes PID to =~/.config/pi-bridge/bridge.pid=.
- Survives pi session exit.
- Killed via =kill $(cat ~/.config/pi-bridge/bridge.pid)= or SIGTERM.
- On death: sessions auto-reconnect within ~10s (SSE reconnect respawns bridge).

** Session Management

- Sessions register with =POST /register= and receive events via =GET /events= (SSE).
- Heartbeat: SSE connection keeps =LastSeen= updated on each event + 30s heartbeat comments.
- Stale sessions (no heartbeat for 5 minutes) are auto-removed.
- Topic conflict: =register= returns 409 if another session already claims the same =topicId=.

** Routing

- Inbound Telegram messages are routed by =message_thread_id= (topic ID) to the registered session.
- =topicId: 0= sessions are not accepted by the bridge (extension falls back to direct polling).
- Unroutable messages (no session for topic) get a Telegram reply: "⚠️ No pi session connected."

** Security

- Only messages from =allowedUserId= are processed. All others silently dropped.
- Bridge listens on =127.0.0.1= only — not exposed to the network.
- Bot token stored in config file (recommend =chmod 600=).
